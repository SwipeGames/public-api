# Reverse calls — semantics you must get right

The platform makes four reverse calls to your endpoints during a game session. They move real money, arrive signed, and are retried. Idempotency and the retry/refund rules are the core of a correct integration — read this before implementing any handler.

## The endpoints

| Endpoint   | Method | Purpose                                  |
| ---------- | ------ | ---------------------------------------- |
| `/balance` | GET    | Return the player's current balance      |
| `/bet`     | POST   | Deduct a bet                             |
| `/win`     | POST   | Credit a win (amount may be `"0"`)       |
| `/refund`  | POST   | Reverse a previously failed bet          |

`sessionID` identifies the player/session — it is the value you passed to `createNewGame()`. Use it to look up the wallet.

## Money

All amounts and balances are **decimal strings in the currency's main unit — not cents** (e.g. `"0.90"` = €0.90, `"100.10"` = €100.10; fiat carries up to 2 decimal places), matching `^(0|[1-9]\d*)(\.\d+)?$`. Never run floating-point math on them. If your wallet stores another unit (e.g. integer cents), convert at the boundary with integer/string arithmetic — never `parseFloat`, multiply, or divide as a float — so you don't introduce rounding error.

## Idempotency (the most important rule)

Every reverse call carries a `txID` — a globally unique UUID v4 generated by Swipe Games. **Use `txID` as the idempotency key.** If you have already processed a `txID`, return the *same* successful `200` response again instead of applying it twice. Retries always reuse the same `txID` (and `roundID`).

- Swipe Games enforces `txID` uniqueness for a **rolling 3-month window**. If you need a longer guarantee, key on the composite (`txID` + `roundID`).
- A round (`roundID`) groups a bet → win (→ refund). `roundID` is unique only within a single game, so scope it to the game.
- Echo back **your** transaction id in each response (your side's id), for reconciliation and debugging.

## Retry & refund policy (differs per endpoint)

**Bet** — *not* retried. Any non-`200` declines the round and notifies the player.
- Return **`500` if you want the action to be refundable.** A `500` or a timeout > 5s triggers an automatic refund.
- Other `4xx` errors are **not** refundable.

**Win** — retried until you return `200` (same `txID`/`roundID` each time). A win error does not decline the bet but is shown to the player. No refunds are issued on win. An `amount` of `"0"` is a valid no-win.

**Refund** — retried until `200`. **Always return `200` with a valid body, even if you have no matching transaction** (e.g. the original bet never landed) — set `balance` to your current value (`0` is acceptable). Returning an error just makes it retry forever.

## Timeout

There is a hard **5-second** limit on every reverse call. Keep handlers fast; defer heavy work to async processing.

## Free rounds

During a free-rounds campaign, bet/win requests arrive with `type: "free"` and an `frID`. Handle per the free-rounds docs (typically: do not debit the player's real wallet on a free bet; credit wins normally). If you do not run free-rounds campaigns, every request will carry `type: "regular"` (the only non-`free` value).

Details: https://swipegames.github.io/public-api/free-rounds

## Signature

Every request includes an `X-REQUEST-SIGN` header (HMAC-SHA256 over the canonical request, keyed with your **Integration API key**). Verify it on every call — SDKs do this inside `parseAndVerify*`; manual verification is in `references/manual.md`.

**Capture the raw request body before any JSON body-parser middleware reads or reparses it.** The POST signature is computed over the exact bytes received, so re-serializing the parsed object changes them and breaks verification. (`GET /balance` has no body — its signature is over canonical JSON of the query params.)

Flows & sequence diagrams: https://swipegames.github.io/public-api/swipegames-integration
